## 12. Team workflow — multi-round debate, not isolated polling

**This is the #2 lesson from Session 91 (after CTO role).** It's now in `mistakes.md` under "Team workflow was isolated polling, not multi-round debate".

**Wrong pattern (what I did in Session 91):**
1. Spawn N subagents in parallel, each asked one independent question
2. Collect outputs
3. I synthesize alone

**Why it's wrong:** Agents are treated as a consultant pool. No cross-critique. No "Agent A reviews Agent B's findings". No convergence loop. The synthesis is just my own opinion weighted by whichever agent was most confident.

**Right pattern:**
1. **Round 1 — parallel exploration:** Spawn N agents (usually 3-5), each with a focused question
2. **Round 2 — cross-critique:** Spawn (N-1) "critic" agents to find holes in each other's Round-1 outputs. Pass Agent A's output as context to Agent B-critic, and so on.
3. **Round 3 — synthesis:** Spawn 1 "synthesizer" agent with all Round 1 + Round 2 outputs. Let it produce the convergence.
4. **Only now** draft your conclusion, grounded in the synthesis.

**Exceptions:**
- For trivial lookups (e.g. "find file X"), single agent is fine.
- For non-strategic technical decisions (e.g. "fix this one bug"), skip the team and just do it.
- For strategic / multi-file / architectural decisions, 2+ rounds are mandatory.

**Anti-pattern warnings:**
- Spawning 5 agents with vague prompts and then cherry-picking the one you liked = worse than 1 agent. It's a ceremony of disagreement, not real debate.
- Don't use multi-round for urgency-driven tasks (CEO waiting). Single round + your judgment is fine when velocity matters.

**Rules of thumb:**
- >2 files affected → min 1 cross-critique round
- >3 files affected → min 1 cross-critique + synthesizer
- Architecture decision → full 3 rounds
- "Ship fast" instruction from CEO → 1 round max, skip to action

**TASK-PROTOCOL v7.0** (`feedback_task_protocol.md`) formalizes this as "RECURSIVE CRITIQUE" — SEC/ARCH/PERF/UX agents must counter each other, not perform consensus.

## 14. CEO escalation rule — both, not either

**This is the #3 lesson from Session 91.** It's now in `mistakes.md` under "CEO escalation worst quadrant: neither solved nor consulted team".

**The conflict:**
- `feedback_autonomy.md` says: "solve yourself first, 3 attempts before escalation"

**Right interpretation (new rule):**

1. **Broadcast to team via subagents** (guardrail-auditor / a11y-scanner / code-reviewer / whichever is relevant) — parallel Round 1 to get 3-5 perspectives

Проблема: [1 line]
Я попробовал: [what, why it failed]

C) [option] — [1-line tradeoff]
Моя рекомендация: [A/B/C] потому что [reason]
Что выбираешь?

This is a CEO-friendly format: short, structured, action-oriented, explicit recommendation.

**When NOT to escalate yet (single-attempt rule):**
If the first attempt revealed a clear path and you're mid-implementation, finish it. Don't escalate mid-fix. Only escalate when you're actually stuck, not just working.
