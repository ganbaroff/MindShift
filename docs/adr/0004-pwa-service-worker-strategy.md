# ADR 0004 — PWA Service Worker Strategy

**Date:** 2026-03-11
**Status:** Accepted
**Author:** Claude (Sprint 6 audit)

---

## Context

MindShift is deployed as a Progressive Web App — users can install it to their home screen on iOS and Android and expect it to open instantly, with or without network. The service worker (SW) is the key enabler of offline capability and perceived performance.

### Requirements

1. **Instant shell load** — app shell (HTML + JS + CSS) must load without network after first visit
2. **Offline task capture** — users must be able to add tasks while offline
3. **Fresh deploys propagate** — when Vercel deploys a new build, users should get the update within one session (not days later due to aggressive caching)
4. **API calls not cached** — Supabase REST calls must always go to the network (stale task data is worse than a short spinner for ADHD users)
5. **Push notifications** — future: reminder support via Web Push

---

## Decision

Use **Vite PWA plugin** (`vite-plugin-pwa`) with **`injectManifest` strategy** (custom SW file at `src/sw.ts`).

### Why `injectManifest` over `generateSW`

| | `generateSW` | **`injectManifest`** |
|---|---|---|
| Control | Fully auto-generated | Custom SW with precache injection |
| Push notifications | Not supported | Full Web Push API access |
| Custom routing logic | Not supported | Full Workbox routing available |
| Complexity | Minimal | Moderate |

MindShift needs push notifications in a future sprint and custom offline fallback logic, making `injectManifest` the correct long-term choice.

### Caching strategy

```
Route                        Strategy            TTL
──────────────────────────── ─────────────────── ───────
/assets/*.{js,css,woff2}    CacheFirst           1 year (content-hashed)
/icon-*.png, favicon.ico    CacheFirst           30 days
/                           NetworkFirst         5 s timeout → cache fallback
/api/*, supabase.co/**      NetworkOnly          never cached
Push notifications           Background Sync API  (Sprint 7)
```

### `registerType: 'autoUpdate'`

The SW registration uses `autoUpdate`, meaning the new SW installs and activates without requiring the user to manually refresh. Combined with Vite's content-hashed filenames, this ensures:
- Old SW serves old assets until all tabs close
- New SW activates on next page load
- No stale JS/CSS mismatch — each hash is unique

### Offline fallback

When the user opens the app offline and the shell is not in cache (first visit), the SW serves a minimal offline shell from the precache manifest. The offline shell shows a "You're offline — tasks you add will sync when you reconnect" message using the same UI pattern as the `ContextRestore` overlay.

---

## Consequences

### Positive
- **~0ms shell load after first visit**: precached assets served from Cache Storage
- **Fine-grained control**: custom SW means we can add push, background sync, and periodic sync as needed
- **Vite integration**: manifest injection is handled by `vite-plugin-pwa`; no manual SW maintenance
- **iOS compatible**: `vite-plugin-pwa` generates the required `apple-touch-icon` and `apple-mobile-web-app-*` meta tags

### Negative / Trade-offs
- **SW complexity**: `injectManifest` requires maintaining `src/sw.ts`; bugs in the SW are hard to debug
- **autoUpdate UX**: if a user has the app open when a deploy happens, their next navigation may cause a brief flash as the new SW activates. Mitigated by showing a "New version available" toast (planned)
- **Safari SW limitations**: background sync and push are limited on iOS; accepted limitation at current maturity

---

## Alternatives Considered

| Alternative | Why rejected |
|---|---|
| `generateSW` (Workbox default) | No push notification support; insufficient routing control |
| Manual SW (no Vite plugin) | Too much boilerplate; precache manifest generation would be manual |
| No SW / no PWA | Loses instant load + offline task capture — core ADHD UX requirement |
| Next.js + next-pwa | Would require migrating away from Vite; disproportionate cost |
