## SECTION 1: ONBOARDING & HEALTH PROFILE

### A. Summary

1. **Onboarding is ultra-short (3 screens, ~2 min)** — Intent → Energy → ADHD Signal. Aligns well with research requirement for minimal cognitive load on first launch.
2. **All questions use visual emoji cards** — zero text fields, zero clinical labels. Fully matches "closed questions with visual cards" requirement.
3. **ADHD subtype collected via behavioural scenario** ("Do you forget tasks out of sight?") — maps to `cognitiveMode` (focused/overview), not clinical diagnosis. Good, but only captures one ADHD dimension (object permanence).
4. **Energy level captured during onboarding** (5-point emoji scale) — affects timer presets, XP multipliers, mascot state. Strong implementation.
5. **Sleep, medication, circadian rhythm — completely absent.** No questions, no progressive profiling, no contextual prompts.
6. **Timer preference not asked** — app is countdown-only. Research suggests count-up/stopwatch is less anxiety-triggering for some ADHD subtypes. No user choice offered.
7. **Progressive profiling is minimal** — RecoveryProtocol (72h+) and ContextRestore (30-72h) exist but don't collect new profile data. No post-session questions, no "Low Battery" contextual prompts.
8. **Shame-free language is excellent** — zero instances of "lazy", "procrastination", "failure", "overdue". Identity-affirming copy throughout ("consistent returner, not a perfect one").

## SECTION 2: SOCIAL LAYER & VIRTUAL COWORKING

### A. Summary

1. **MindShift is 100% single-user.** Zero social/multiplayer features exist in production code.
2. **No social tables in Supabase** — no friends, rooms, invitations, presence. All tables enforce user isolation via RLS.
3. **No Model A (1:1 sessions)** — no partner matching, no shared timer, no abstract presence.
4. **No Model B (small rooms)** — no room creation, no status-emoji system, no 2-4 person coworking.
5. **No Model C (Orbit/ambient presence)** — no heatmap, no "sky of focusers", no WebSocket presence.
6. **No Model D (AI body-double)** — Mochi mascot exists as emotional anchor but has zero accountability features. It's a visual state indicator, not a co-regulator.
7. **No leaderboards, no public streaks, no social comparison** — this is BY DESIGN per shame-free principles. Excellent RSD protection.
8. **No chat or messaging of any kind** — consistent with the "no free chat in focus spaces" requirement, but only because there are no focus spaces at all.

## SECTION 3: LONG-TERM RHYTHMS & BURNOUT PREVENTION

### A. Summary

1. **No seasonal/year modes** (Launch/Maintenance/Recovery/Sandbox). App has static `appMode` (minimal/habit/system) set once during onboarding. No temporal rotation.
2. **Burnout data is collected but never analyzed.** `user_behavior` table tracks snooze count, completion ratio, session duration — but no burnout score, no trend detection, no alerts.
3. **Energy-based planning is partial** — energy level affects timer duration and XP multipliers, but no Traffic Light task classification, no Two-Thirds rule, no hard task quotas.
4. **Zero sleep/HRV/wearable integration.** User self-report energy only.
5. **Reduced Stimulation mode exists** — desaturates colors, disables glow/confetti. A "soft recovery mode" but doesn't simplify features or reduce task load.
6. **Streaks are explicitly zero** — exemplary shame-free design. Cumulative progress (`completedTotal`) used internally only, never displayed as streak.
7. **Statistics are weekly only, growth-framed** — "Perfect week" / "You showed up!" / "Fresh start ahead". No monthly/quarterly. No burnout trajectory.
8. **No quarterly rituals** — no monthly review, no Hyperfocus Autopsy, no seasonal check-in. App lives in the present moment by design.
9. **AI mentor is reactive only** — RecoveryProtocol triggers after 72h+ silence. No proactive burnout detection, no "you seem burned out" intervention.
10. **RecoveryProtocol (72h+) and ContextRestore (30-72h) are excellent** — full-screen shame-free re-entry with AI messages, micro-win suggestions, task archival.

| Area | Status | Comment |
